# Hermes Continuation v0.3.0 — 自動觸發策略執行計畫

> **版本：** Draft v1 | **日期：** 2026-05-12 | **基於：** v0.2.0（5 CLI + 5 plugin tools）
>
> **一句話目標：** 不等人手動下 `/handoff`，Hermes 自己在該交接的時候主動提醒。

---

## 核心問題

v0.2.0 的 `watch` 很好，但需要**人主動呼叫**──人在對話中、終端機裡打指令、或 plugin tool 被手動觸發。真正有價值的情境是：

> 你跟 Hermes 連續開發了 45 分鐘，call 了 80 次 tool，改了 12 個檔案──但你忘了要交接。Hermes 應該在這個時候自己跳出來說「該交接了」。

v0.3.0 要做的就是把這個「被動」變成「主動」。

### 三個觸發來源

| 來源 | 機制 | 適合場景 |
|------|------|----------|
| **Gateway Session** | Hermes gateway 在每個 turn 結束時檢查條件，自動呼叫 watch | 即時對話中的主動提醒 |
| **Cron 定時** | 每 N 分鐘掃描指定 repo，判斷是否需要提醒 | 跨對話、背景監控 |
| **Context Monitor** | 從 Hermes runtime 拿真實的 tool-call 數、session 時長 | 取代手動輸入 `--tool-calls 8` |

三者可以獨立運作，也可疊加──Gateway session 即時提醒 + Cron 定時掃描備援。

## 任務拆解

### Phase 3A：Context Monitor — 自動收集真實數據

**問題：** v0.2.0 的 watch 需要手動傳 `--tool-calls 8 --elapsed-minutes 45`。人根本不知道確切數字。

**解法：** 寫一個 Context Monitor 模組，從 Hermes runtime session 自動撈取真實數據。

#### Task 3A-1：`context_monitor.py` — 收集 session metrics

| 欄位 | 內容 |
|------|------|
| **檔案** | `src/hermes_continuation/context_monitor.py` |
| **依賴** | 無外部依賴，讀 Hermes session state |
| **輸出** | `{"tool_calls": 82, "elapsed_minutes": 47, "changed_files": 12, "session_id": "..."}` |

**實作要點：**
- 從 `collect_git_state()` 拿真實的 changed_files
- 從 Hermes session metadata（如果可存取）拿 tool-call count 和 session 時長
- 如果 session metadata 不可存取，提供 hook 讓 gateway/caller 注入這些數值
- 向後兼容：CLI 的 `--tool-calls` / `--elapsed-minutes` 手動值仍然優先（manual overrides auto）

**驗收標準：**
```bash
python -m pytest -q tests/test_context_monitor.py  # 新增測試
# 在 git repo 中跑 → 拿到真實的 changed_files
# 手動傳入值 overrides 自動值
```

---

#### Task 3A-2：整合 Context Monitor 到 Watch

| 欄位 | 內容 |
|------|------|
| **改動** | `watch.py` — `build_watch_result()` 改用 context_monitor 自動填充 |
| **行為** | 如果 caller 沒傳 `tool_calls`，自動從 context_monitor 拿；拿不到就留 None |

**改動範圍：** ~20 行，不破壞現有 API

**驗收標準：**
- 現有 watch tests 全部通過（沒有 regression）
- watch 在 git repo 中自動抓到 changed_files，不再需要手動傳

### Phase 3B：Gateway 自動觸發

**問題：** Arthur 在 Feishu/Telegram 上跟 Hermes 對話時，完全不會想到要打 `/handoff watch`。

**解法：** 在 Hermes Gateway 的 session lifecycle 中埋入自動觸發點。

#### Task 3B-1：Gateway Session Hook — 回合結束時自動檢查

| 欄位 | 內容 |
|------|------|
| **機制** | Gateway 在每個 user turn 結束、agent response 送出後，異步呼叫 watch |
| **觸發條件** | tool_calls > 5 **且** elapsed > 30min **且** 過去 20 分鐘內沒推送過 |
| **通知方式** | 只在 `advise` 或 `prepare` 時才推送，`observe` 保持安靜 |

**設計要點：**

1. **非阻塞** — watch 在背景跑，不卡住下一個 user turn
2. **去重** — 用 `_last_watch_notification` 追蹤，20 分鐘內不重複推送
3. **只在有東西時說** — `observe` 不推送（避免噪音）
4. **安全** — block 結果發送簡短警告，不含機敏資訊

**配置方式（`config.yaml`）：**
```yaml
plugins:
  enabled:
    - hermes-continuation
  config:
    hermes-continuation:
      auto_watch:
        enabled: true
        tool_calls_threshold: 5
        elapsed_minutes_threshold: 30
        cooldown_minutes: 20
        notify_on: ["advise", "prepare", "block"]
```

**驗收標準：**
- Gateway session 跑 40 分鐘 + 超過 5 tool calls → 自動收到 watch 提醒
- `observe` 結果不推送
- 20 分鐘內不重複推送
- 可以透過 config 關閉整功能

---

#### Task 3B-2：Feishu 報告群通知

| 欄位 | 內容 |
|------|------|
| **目標** | `oc_3dcfe36103c714ac0c7736d7f1725f95`（Feishu 報告群） |
| **內容** | 簡短摘要：「Repo X 已經開發 45 分鐘，改了 12 個檔案，建議交接」 |

**設計要點：**
- 通知格式簡短、白話、非技術化
- 不要貼整個 watch output（太長在手機上不好讀）
- 包含一個可以點或複製的 `/handoff prepare` 指令

**驗收標準：**
- 自動觸發時 Feishu 報告群收到簡短提醒
- 格式在手機上可讀

### Phase 3C：Cron 定時 Watch

**問題：** 你沒在對話中（睡覺、開會、接送小孩），但 codebase 有大量未 commit 變更──你回來時早忘了做到哪。

**解法：** 用 Hermes cronjob 定期掃描指定 repo。

#### Task 3C-1：Cron Watch 腳本

| 欄位 | 內容 |
|------|------|
| **腳本** | `~/.hermes/scripts/handoff_watch_cron.py` |
| **排程** | 每 30 分鐘 |
| **行為** | 掃指定 repo → 呼叫 watch → 只在 advise/prepare/block 時輸出 |
| **通知** | Feishu 報告群 |

**腳本邏輯：**
```python
import sys
sys.path.insert(0, "/home/zycas/hermes-continuation/src")
from hermes_continuation.watch import build_watch_result

# 掃描指定 repos
repos = [
    "/home/zycas/hermes-continuation",
    "/home/zycas/.hermes/hermes-agent",
]
for repo in repos:
    result = build_watch_result(repo, auto_task_state=True)
    if result["level"] in ("advise", "prepare", "block"):
        print(f"{repo} → {result['level']}")
```

**Cron job 配置：**
```
cronjob create:
  name: "handoff-watch-cron"
  script: "handoff_watch_cron.py"
  schedule: "every 30m"
  model: {"model": "glm-4.7", "provider": "ollama-cloud"}
  prompt: "腳本已執行完畢。若有輸出，摘要發送 Feishu。無輸出回覆 SILENT。"
  deliver: "local"
  enabled_toolsets: ["terminal"]
```

**驗收標準：**
- 手動 `cronjob run` → repo 有未 commit 變更時收到 Feishu 提醒
- `observe` 時安靜（SILENT）
- 不重複提醒同一組變更（透過 git commit hash 追蹤）

### Phase 3D：觸發閾值調校（Dogfood 驅動）

**問題：** 預設 `tool_calls > 5`、`elapsed > 30min` 是猜的，不知道實際好不好用。

**解法：** 在 dogfood 期間收集數據，根據真實使用情況調整。

#### Task 3D-1：Watch 日誌記錄

每次 watch 執行時記錄到本地 log（不經 LLM、零 token 成本）：

```json
{"ts": "2026-05-15T14:30:00", "repo": "hermes-continuation", "level": "advise",
 "tool_calls": 8, "elapsed": 45, "changed_files": 3, "trigger": "gateway"}
```

#### Task 3D-2：兩週回顧 + 調校

| 收集指標 | 決策 |
|----------|------|
| 觸發頻率 | 每天幾次？太頻繁 → 提高閾值 |
| false positive 率 | 幾次 advise 但用戶沒理？→ 降低敏感度 |
| 錯過率 | 用戶手動 `/handoff` 時 watch 還在 observe？→ 降低閾值 |

## 風險與緩解

| 風險 | 影響 | 緩解 |
|------|------|------|
| Gateway 背景 watch 拖慢回應 | 使用者感受延遲 | 非阻塞異步執行；可設定 timeout |
| 太頻繁推送造成噪音 | 使用者關掉功能 | 去重機制 + 可調 cooldown |
| Context Monitor 拿不到 session metadata | 無法自動填充 | 保留手動 fallback；標記 `source: "manual"` vs `source: "auto"` |
| Cron 掃描到未 push 的變更 | 重複提醒同一組變更 | 追蹤 last_notified_commit |

---

## 不做的（Non-goals for v0.3.0）

- ❌ 不自動 `create` 交接包（永遠只 advise/prepare）
- ❌ 不啟動 daemon 背景進程（只用 cron + gateway hook）
- ❌ 不解析完整 Hermes 對話記錄
- ❌ 不修改 Hermes 核心程式碼
- ❌ 不做跨機器雲端同步

這些是 v0.4+ 的事。
