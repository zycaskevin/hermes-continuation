# Hermes Continuation v0.3.0 — 自動觸發策略執行計畫

> **版本：** Draft v2（Kanban 審查修訂） | **日期：** 2026-05-12 | **基於：** v0.2.0（5 CLI + 5 plugin tools）
>
> **一句話目標：** 不等人手動下 `/handoff`，Hermes 自己在該交接的時候主動提醒。

---

## 核心問題

v0.2.0 的 `watch` 很好，但需要**人主動呼叫**──人在對話中、終端機裡打指令、或 plugin tool 被手動觸發。真正有價值的情境是：

> 你跟 Hermes 連續開發了 45 分鐘，call 了 80 次 tool，改了 12 個檔案──但你忘了要交接。Hermes 應該在這個時候自己跳出來說「該交接了」。

v0.3.0 要做的就是把這個「被動」變成「主動」。

### 三個觸發來源

| 來源 | 機制 | 適合場景 |
|------|------|----------|
| **Gateway Wrapper** | 在 Gateway 呼叫 Hermes 的外部包裝層，每次 response 後手動呼叫 `hermes_handoff_watch` | 即時對話中的主動提醒 |
| **Cron 定時** | 每 N 分鐘掃描 config 中列出的 repo，判斷是否需要提醒 | 跨對話、背景監控 |
| **Context Monitor** | 自動收集 git changed_files；接收外部注入的 tool-call / elapsed 數值 | 取代手動輸入 `--tool-calls 8` |

三者可以獨立運作，也可疊加──Gateway wrapper 即時提醒 + Cron 定時掃描備援。

### Task A：Context Monitor — 自動收集真實數據

**問題：** v0.2.0 的 watch 需要手動傳 `--tool-calls 8 --elapsed-minutes 45`。人根本不知道確切數字。

**解法：** 寫一個 Context Monitor 模組，自動收集 git changed_files，並提供 hook 讓外部注入 session metrics。

#### Task A-1：`context_monitor.py` — 收集可用數據

| 欄位 | 內容 |
|------|------|
| **檔案** | `src/hermes_continuation/context_monitor.py` |
| **依賴** | 複用現有 `collect_git_state()` |
| **輸出** | `{"tool_calls": 82, "elapsed_minutes": 47, "changed_files": ["src/foo.py"], "source": "auto"}` |

**數據來源策略（由易到難）：**

| 數據 | 來源 | 可靠性 |
|------|------|:---:|
| `changed_files` | `collect_git_state()` → 已驗證可用 | ✅ 確定 |
| `tool_calls` | 外部注入（Gateway/caller 提供） | ⚠️ 被動 |
| `elapsed_minutes` | 外部注入（Gateway/caller 提供） | ⚠️ 被動 |
| session metadata | 嘗試從 Hermes runtime 讀取 | ❓ 待 Phase 0 驗證 |

**核心設計原則：**

- **永遠有 fallback：** 拿不到的數據留 `None`，不捏造
- **手動值優先：** CLI 的 `--tool-calls` / `--elapsed-minutes` 永遠 override 自動值
- **標記來源：** `"source": "auto"` vs `"source": "manual"` 讓 user 知道數據從哪來
- **零副作用：** 只讀不寫，不記錄對話內容

#### Task A-2：整合 Context Monitor 到 Watch

| 欄位 | 內容 |
|------|------|
| **改動** | `watch.py` — `build_watch_result()` 加一個 `context` 參數接收外部數據 |
| **行為** | 如果 caller 沒傳 `tool_calls`/`elapsed`，從 `context` dict 取值；都沒有的話留 None |

**改動範圍：** ~30 行，不破壞現有 API。

#### 測試策略

```bash
# test_context_monitor.py
test_collects_changed_files_in_git_repo       # git repo 中拿到真實 changed_files
test_degrade_non_git_directory                 # 非 git 目錄回空 changed_files
test_manual_values_override_auto               # 手動值覆蓋自動值
test_source_field_marks_auto_vs_manual         # source 欄位正確標記
test_never_collects_conversation_content       # 確保不讀對話內容
```

### Task B：Gateway 自動觸發

**問題：** Arthur 在 Feishu/Telegram 上跟 Hermes 對話時，完全不會想到要打 `/handoff watch`。

**解法：** 在 Gateway 的外部包裝層（不修改 Hermes 核心）手動呼叫 watch plugin tool。

#### Task B-1：Gateway Wrapper — 明確的呼叫合約

> **重要：** 不假設 Gateway 有內建 hook。採「包裝層手動呼叫」模式。

**合約：** Gateway 在送出 agent response 後，**手動**呼叫以下函數：

```python
from hermes_continuation.plugin import hermes_handoff_watch
import json

# 在 Gateway 的 response 送出之後：
result = json.loads(hermes_handoff_watch({
    "repo_path": str(working_dir),
    "goal": current_goal or "",
    "next_task": next_task or "",
    "tool_calls": session_tool_call_count,    # Gateway 自己計數
    "elapsed_minutes": session_elapsed_minutes, # Gateway 自己計時
    "auto_task_state": True,
}))

if result.get("level") in ("advise", "prepare", "block"):
    # 發送通知（格式見 Task B-2）
```

**Gate 條件（全部滿足才推送）：**

1. `tool_calls` >= 閾值（預設 5）
2. `elapsed_minutes` >= 閾值（預設 30）
3. 距離上次推送 >= cooldown（預設 20 分鐘）
4. `level` 不是 `observe`
5. config 中 `auto_watch.enabled: true`

**配置方式（`~/.hermes/config.yaml`）：**
```yaml
plugins:
  config:
    hermes-continuation:
      auto_watch:
        enabled: true
        tool_calls_threshold: 5
        elapsed_minutes_threshold: 30
        cooldown_minutes: 20
        notify_levels: ["advise", "prepare", "block"]
```

#### Task B-2：通知格式

**目標：** Feishu 報告群。通知必須簡短、手機可讀、不洩漏專案細節。

**格式設計：**
```
⚠️ 有一個開發中的專案建議交接
已開發約 45 分鐘，使用 80+ 次工具，12 個檔案有變更
→ 回對話中輸入 /handoff prepare 來預覽交接內容
```

**設計原則：**
- ❌ 不顯示具體 repo 名稱或路徑（防止群組內資訊洩漏）
- ❌ 不顯示變更的檔案名稱
- ✅ 顯示約略數字（45 分鐘、80+ 次、12 個檔案）
- ✅ 給出下一步行動指引（`/handoff prepare`）
- ✅ 在手機上可以一眼讀完

#### 測試策略

```bash
# test_auto_watch.py（新增）
test_observe_does_not_trigger_notification       # observe → 無推送
test_advise_triggers_notification                 # advise → 推送
test_cooldown_prevents_duplicate_notifications    # cooldown 去重
test_config_disabled_suppresses_all               # enabled:false → 完全靜默
test_block_level_does_not_leak_secrets             # block 不含機敏資訊
```

### Task C：Cron 定時 Watch

**問題：** 你沒在對話中（睡覺、開會、接送小孩），但 codebase 有大量未 commit 變更──你回來時早忘了做到哪。

**解法：** 用 Hermes cronjob 定期掃描 config 中指定的 repo。

#### Task C-1：Cron Watch 腳本

| 欄位 | 內容 |
|------|------|
| **腳本** | `~/.hermes/scripts/handoff_watch_cron.py` |
| **排程** | 每 30 分鐘 |
| **行為** | 從 config 讀取監控清單 → 逐一呼叫 watch → 只在 advise/prepare/block 時輸出簡短摘要 |
| **通知** | Feishu 報告群（格式同 Task B-2） |

**Repo 清單管理：** 不 hardcode 在腳本中，改從 config 讀取：
```yaml
plugins:
  config:
    hermes-continuation:
      watch_repos:
        - /home/zycas/hermes-continuation
        - /home/zycas/.hermes/hermes-agent
```

#### Task C-2：重複提醒防止

追蹤每個 repo 的最後通知 commit hash，同一 commit 不重複提醒：
```python
state = load_state()  # {"repo_path": "last_notified_commit_hash"}
current = get_head_commit(repo)
if current == state.get(repo):
    return  # 同一 commit，跳過
state[repo] = current
save_state(state)
```

#### 測試策略

```bash
# 手動驗證
cronjob run → observe 時安靜（SILENT）✅
cronjob run → advise 時 Feishu 收到通知 ✅
# 同一 commit 第二次 run → 安靜（不重複）✅
```

## Degradation Path（降級路徑）

如果某些技術前提無法滿足：

| 如果... | 則... |
|---------|-------|
| Gateway wrapper 模式不可行（Gateway 環境無法 import plugin） | 自動觸發僅透過 Cron 定時提供；Gateway 中仍需手動 `/handoff watch` |
| Context Monitor 完全拿不到任何自動數據 | 保留現有手動模式；自動觸發退化成定時簡單提醒（「你的 repo 有未 commit 變更」） |
| Cron 不可靠或不穩定 | 僅保留 Gateway wrapper 模式；不做背景掃描 |

---

## Rollback 計畫

如果上線後太吵或出問題：

1. **短期（立即靜默）：** 設定 `auto_watch.enabled: false`，所有自動觸發立即停止。無需降版。
2. **中期（保留手動）：** user 仍可手動 `/handoff watch`、`hermes-handoff watch`，v0.2.0 功能不受影響。
3. **長期（徹底移除）：** 移除 Task B/C 的程式碼和 config，退回純 v0.2.0。

## 安全紅線（Auto-Trigger Safety Boundary）

> 本章節同時應寫入 `TRIGGER_POLICY.md`。

### 自動觸發可以做的事

| 操作 | 條件 |
|------|------|
| 讀取 git state（branch、commit、changed_files） | 僅在 config 指定的 repo 中 |
| 接收外部注入的 session metrics（tool_calls、elapsed） | 由 Gateway/caller 主動提供，不自行存取 |
| 寫入本地 watch 日誌（level、數字、時間戳） | 不含對話內容、檔案內容、repo 路徑 |
| 發送通知到 Feishu 報告群 | 簡短摘要、不含 repo 名稱 |

### 自動觸發不可以做的事

| 操作 | 原因 |
|------|------|
| 讀取對話內容或 user message | 隱私邊界 |
| 讀取檔案內容（非 git metadata） | 可能包含機敏資訊 |
| 自動呼叫 `create` 寫出交接包 | 永遠只 advise/prepare |
| 將數據送出本機（除 Feishu 通知外） | 資料不外洩 |
| 在 user 未同意的情況下記錄 session 內容 | 隱私優先 |

---

## 不做的（Non-goals for v0.3.0）

- ❌ 不自動 `create` 交接包（永遠只 advise/prepare）
- ❌ 不啟動 daemon 背景進程（只用 cron + gateway wrapper）
- ❌ 不解析完整 Hermes 對話記錄
- ❌ 不修改 Hermes 核心程式碼
- ❌ 不做跨機器雲端同步

這些是 v0.4+ 的事。
