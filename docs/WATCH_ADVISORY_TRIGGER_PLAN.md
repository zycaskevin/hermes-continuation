# Watch / Advisory Auto-Trigger Implementation Plan

## Goal

Add an optional `hermes-handoff watch` command as a read-only/advisory sidecar surface. The command evaluates local signals and prints or delivers a recommendation or prepare preview, but it never writes handoff packets and never creates `.hermes/handoffs/` output.

The first version should help users notice when a handoff would be useful without changing the existing boundary:

- `doctor` recommends.
- `prepare` previews.
- `create` writes only when explicitly invoked by the user.
- `watch` orchestrates local signal evaluation and routes to recommendation/preview output; it is not a new write path.

## Non-goals

This plan does not include:

- Hermes core changes.
- Automatic session restart or fresh-agent launch.
- Full Hermes transcript parsing.
- Hidden `create` invocation or hidden packet writes.
- A daemon/background loop by default.
- Cloud sync, dashboard UI, or remote telemetry.
- Committing generated artifacts such as `.hermes/handoffs/`, `graphify-out/`, caches, or package build output.

## Design

`watch` should be a thin wrapper/orchestrator over existing public sidecar behavior:

1. Normalize explicit CLI signal inputs into a small watch signal model.
2. Reuse `evaluate_handoff_recommendation()` from `src/hermes_continuation/doctor.py` for recommendation level, reasons, and safety-sensitive decision behavior.
3. Reuse `build_prepare_preview()` from `src/hermes_continuation/prepare.py` when the evaluated level and available fields support a read-only preview.
4. Render either human text or JSON with the same level vocabulary as advisory/prepare surfaces: `observe`, `advise`, `prepare`, `block`.
5. Never call create-path packet writers from watch.

The watch implementation should prefer composition over duplicate policy logic. If new signal interpretation is needed, add a helper module or a carefully scoped helper near `doctor`; do not create an independent decision engine that can drift from `doctor`/`prepare`.

## Initial MVP Scope

### One-shot command only

Implement one-shot evaluation first:

```bash
hermes-handoff watch --repo . --goal "Finish release QA" --next "Run final smoke" [--json]
```

Behavior:

- collect local repo/task signals from explicit arguments and safe existing helpers;
- evaluate once;
- print a recommendation or prepare preview;
- exit with no background loop and no writes.

Do not add `--interval` or long-running monitoring in the MVP. A future `--interval` can be considered only after the one-shot CLI contract proves useful and quiet enough.

### Optional signal inputs

Support explicit optional inputs so tests and callers can model local state without transcript parsing:

- `--tool-calls <int>`: number of tool calls observed by the caller.
- `--elapsed-minutes <int>`: approximate task/session duration.
- `--dirty-threshold <int>`: changed-file threshold that increases recommendation strength.
- `--not-run <text>`: repeatable verification gate not run.
- `--failing <text>`: repeatable failing verification gate.
- `--verified <text>`: repeatable passed verification gate.
- `--explicit-request`: user explicitly asked for a handoff/continuation checkpoint.

Use conservative defaults. Missing `goal` or `next` must degrade to `advise`, not fabricate `prepare` state.

### Output levels

`watch` output must use these levels consistently:

| Level | Meaning | Watch behavior |
| --- | --- | --- |
| `observe` | Signals are below recommendation threshold. | Print concise status in CLI; JSON includes reasons/signals. |
| `advise` | A handoff may be helpful, but required preview state is incomplete or threshold is advisory only. | Print reasons and suggested next action; no create command unless generated by existing safe preview behavior. |
| `prepare` | Known safe state supports a read-only prepare preview. | Print or return the `build_prepare_preview()` envelope with `would_write: false`. |
| `block` | Safety gate prevents a safe preview/write recommendation. | Print blocker category and remediation without raw secret values or sensitive paths. |

## Safety Gates

`watch` must be safe by construction:

- Read-only: no Markdown/JSON packet writes, no `.hermes/handoffs/` creation, no commits, no generated artifacts committed.
- No hidden create: `watch` must never call the create handler or packet-writing functions.
- Private-key and severe sensitive-value findings force `block`.
- Raw secret values and secret-looking repo paths must not be printed in human or JSON output.
- Safety blockers suppress any safe create command.
- Generated/runtime artifacts remain ignored/local and should not be staged.
- All examples must use obvious fake values or `[REDACTED]` placeholders.

## Task Breakdown

### Task 1 — Watch signal model/helper

- Add a small signal model/helper module, for example `src/hermes_continuation/watch.py`, or extend `doctor` carefully if that keeps policy centralized.
- Convert CLI values into the existing recommendation inputs expected by `evaluate_handoff_recommendation()`.
- Include signal provenance in JSON output so users can see whether a level came from tool count, elapsed time, dirty repo state, failing/not-run gates, explicit request, or safety blockers.
- Avoid duplicating the full recommendation matrix.

### Task 2 — CLI parser/handler

- Add `watch` to the CLI parser beside `create`, `resume`, `doctor`, and `prepare`.
- Accept `--repo`, `--goal`, `--next`, `--json`, and the MVP signal flags.
- Default to human-readable output; support JSON for integrations.
- Keep exit behavior conservative: normal advisory levels exit successfully; argument errors or internal failures exit non-zero.
- Do not add plugin/gateway commands in the MVP.

### Task 3 — Tests

Add focused tests for:

- `observe`: low signals, clean/safe state, no writes.
- `advise`: elevated signals or missing required preview state, no fabricated goal/next.
- `prepare`: explicit request plus complete safe goal/next returns a prepare preview with `would_write: false`.
- `block`: private-key or sensitive path/value cases block without printing raw secret data.
- No-write behavior: `watch` never creates `.hermes/handoffs/` and never emits packet files.
- JSON shape: includes `level`, `signals`, `recommendation` or `preview`, `would_write: false`, and no raw secret paths.

Suggested test files:

- `tests/test_watch.py` for helper/model and no-write behavior.
- `tests/test_cli_watch.py` for CLI parser, human output, JSON output, and exit behavior.
- Extend `tests/test_public_docs.py` only if public docs are updated in the implementation task.

### Task 4 — Docs updates

When implementation starts, update public docs only after the CLI contract is stable:

- `README.md`: add a short read-only `watch` example.
- `docs/USAGE.md`, `docs/USAGE.zh-TW.md`, `docs/USAGE.zh-CN.md`: describe one-shot watch, no-write behavior, and non-goals.
- `docs/AUTOMATIC_HANDOFF_TRIGGER_POLICY.md`: record that one-shot watch is the first auto-trigger implementation surface.
- `docs/PLUGIN_WRAPPER.md`: mention no plugin/gateway watch command yet, if relevant.
- `PROGRESS.md`: mark implementation status accurately.

### Task 5 — Runtime smoke/verification

- Run CLI watch in a temporary git repo for human and JSON output.
- Verify `.hermes/handoffs/` is absent after watch.
- Verify `prepare` and `doctor` still behave as before.
- If Graphify rebuild changes generated graph files, leave them uncommitted unless explicitly requested.

## Acceptance Criteria

Implementation is complete only when all of the following are true:

- `hermes-handoff watch` exists as a one-shot CLI command.
- `watch` supports `--repo`, `--goal`, `--next`, `--json`, `--tool-calls`, `--elapsed-minutes`, `--dirty-threshold`, repeatable `--not-run`, repeatable `--failing`, repeatable `--verified`, and `--explicit-request`.
- Output levels are exactly `observe`, `advise`, `prepare`, and `block`.
- `watch` reuses `evaluate_handoff_recommendation()` and `build_prepare_preview()` rather than implementing a separate decision engine.
- `watch` never writes packets, never creates `.hermes/handoffs/`, and never invokes hidden `create` behavior.
- Missing `goal` or `next` degrades to `advise` unless blocked by safety.
- Private-key or sensitive-value cases return `block` without raw secret values or raw secret-looking paths.
- Public docs clearly state the read-only boundary and non-goals.
- Generated artifacts are not staged or committed.

### Secret scan

```bash
python - <<'PY'
import pathlib
import re
import sys

root = pathlib.Path('.')
include_suffixes = {'.py', '.md', '.toml', '.yaml', '.yml', '.json', '.txt'}
skip_dirs = {'.git', '.hermes', '_knowledge_base', 'graphify-out', '.pytest_cache', '__pycache__', 'caches'}
patterns = [
    re.compile(r'(?i)(api[_-]?key|secret|token|password)\s*[:=]\s*["\'](?!\[REDACTED\])[A-Za-z0-9_\-]{8,}["\']'),
    re.compile(r'-----BEGIN (?:RSA|OPENSSH|EC|DSA) PRIVATE KEY-----'),
    re.compile(r'ghp_[A-Za-z0-9]{20,}'),
    re.compile(r'sk-[A-Za-z0-9]{16,}'),
]
failures = []
for path in root.rglob('*'):
    if not path.is_file():
        continue
    rel = path.relative_to(root)
    rel_str = rel.as_posix()
    if set(rel.parts) & skip_dirs:
        continue
    if rel_str.endswith('.egg-info') or '/.egg-info/' in rel_str:
        continue
    if path.suffix not in include_suffixes:
        continue
    if not (
        rel_str.startswith('src/')
        or rel_str.startswith('tests/')
        or rel_str.startswith('docs/')
        or rel_str.startswith('.github/workflows/')
        or rel_str in {'README.md', 'PROGRESS.md', 'pyproject.toml', 'CHANGELOG.md'}
    ):
        continue
    text = path.read_text(encoding='utf-8', errors='ignore')
    for pattern in patterns:
        if pattern.search(text):
            failures.append(rel_str)
            break
if failures:
    print('Potential secret findings:')
    for item in sorted(set(failures)):
        print(f' - {item}')
    sys.exit(1)
print('Lightweight secret scan passed: 0 findings.')
PY
```

## Risks

- Threshold tuning may be noisy or too quiet; start conservative and make defaults easy to adjust.
- False positives can annoy users during short tasks; one-shot mode avoids unsolicited prompts at first.
- Duplicate logic could drift from `doctor`; reuse the existing evaluator and preview builder.
- Users may confuse `watch` with `create`; documentation and output must repeat that watch is read-only.
- Feishu/Telegram readability may require concise human output and minimal tables.

## Recommended Start

Implement one-shot CLI `watch` first. Do not add plugin/gateway commands, background loops, or `--interval` until the CLI contract, output shape, and no-write safety behavior are stable.
